---

## 1. Why this doc exists

The Relocation Systems Map (D-01) hinges on identifying which clusters of the client's life count as *systems* worth tracking. The dictionary definition of "system" is operationally useless — too generic to discriminate. This note provides a Hekswerk-specific working definition, a six-criterion checklist for in-the-moment use, worked examples, and the adjacent distinctions that keep the category clean.

Reading time: ~10 minutes end-to-end. Reference time once internalized: under 30 seconds for a single judgment call.

---

## 2. Working definition

> A *system* is a coherent cluster of interconnected concerns within a relocation that has its own internal logic, persists across multiple decisions over time, and has characteristic failure modes if mishandled. It is what professional specialists exist around when the cluster matures enough — and what well-intentioned generalists trip on when they treat it as a one-off task.

This definition is operational, not philosophical. It exists to discriminate the things that belong on the Map from the things that don't.

---

## 3. The six-criterion checklist

Apply in the moment, mentally or on paper. If a candidate passes 3+, it's probably a system. 5+ and it definitely belongs on the Map.

### 1. Coherence

Multiple sub-elements that belong together because they're about the same domain. A real system has internal parts that hang together — not a single task masquerading as a category.

*Test*: If you split it into sub-pieces, do those sub-pieces obviously belong in the same conversation, or do they fragment into unrelated topics?

### 2. Domain logic

Its own knowledge area you either need to learn or hire for. Real systems have recognizable disciplines around them.

*Test*: If you tried to fake your way through using only generic project management or common sense, would you get burned?

### 3. External gates

Navigating it requires someone or something outside the client's direct control — a government office, a lawyer, a market, a biological constraint, a timing window.

*Test*: Can the client just decide and execute, or does the world need to grant something?

### 4. Connectedness

Load-bearing dependencies on or from other systems. Almost everything in a move touches everything else faintly; the question is whether the connection actually carries weight in sequencing or outcome.

*Test*: If this thing changes — delayed, dropped, modified — does at least one other system have to react?

### 5. Persistence

Generates multiple decision points over time, not just one. A binary one-off is usually a decision living inside a system, not a system itself.

*Test*: Are there still meaningful judgment calls left to make six weeks from now? Six months from now?

### 6. Characteristic failure mode

A specific bad outcome if mishandled, distinct from mishandling other systems. Each system has its own disaster — generic "things go wrong" doesn't count.

*Test*: Can you name the specific way this one breaks, in two words?

## 4. Worked examples

### Clear systems (pass 5–6 criteria)

**Pet transport.**

- Coherence: vet certs + transit method + timing + quarantine + insurance. ✓
- Domain logic: animal welfare, transit regulations, cargo law. ✓
- External gates: vets, transport providers, destination quarantine authorities. ✓
- Connectedness: timing affects travel mode affects housing dates. ✓
- Persistence: decisions span 6+ months. ✓
- Failure mode: animal trauma, denied entry, separation. ✓
- **Verdict**: System. Definitely on the Map.

**Visa pathway (e.g., DAFT).**

- Coherence: business plan + capital deposit + KVK registration + attorney engagement + appointment scheduling. ✓
- Domain logic: immigration law. ✓
- External gates: IND, KVK, attorney, embassy. ✓
- Connectedness: gates almost everything downstream. ✓
- Persistence: months of decisions. ✓
- Failure mode: denial, deportation. ✓
- **Verdict**: System. The spine, usually.

### Edge cases that resolve toward "system"

**Language acquisition.**

The gut heuristic ("damn that's a lot to hold") may not fire — week to week it can feel light. But:

- Coherence: pedagogy + practice + assessment + integration. ✓
- Domain logic: linguistics, learning science, language-specific quirks. ✓
- External gates: native speakers, classes, certification exams if pursued. partial ✓
- Connectedness: affects employment, friendship, healthcare navigation. ✓
- Persistence: years. ✓✓
- Failure mode: isolation, professional ceiling, identity drift. ✓
- **Verdict**: System. The complement test ("is there a professional specialty around this?" — yes, language schools, tutors, integration coaches) confirms it.

**Professional credential recognition.**

Often feels like "just paperwork" until you're in it.

- Coherence: credential evaluation + apostilles + translations + destination-side recognition body + possibly remediation coursework. ✓
- Domain logic: international credential evaluation. ✓
- External gates: evaluation services, destination regulators. ✓
- Connectedness: gates employment in regulated professions. ✓
- Persistence: 6–18 months commonly. ✓
- Failure mode: cannot work in field. ✓
- **Verdict**: System. Easy to miss because clients without regulated professions don't think about it; critical to catch for those who do.

### Things that look like they might be systems but aren't

**Buying a suitcase set.**

- Coherence: limited — basically one product purchase. ✗
- Domain logic: none beyond consumer research. ✗
- External gates: none meaningful. ✗
- Persistence: one decision. ✗
- **Verdict**: Task. Lives inside *Physical Logistics*.

**"Worry about my mom."**

- Coherence: ambiguous — what is the actual cluster? ?
- Domain logic: depends on what the worry actually is. ?
- **Verdict**: Concern, not system. The Phase 1 work is to translate this concern into the system(s) it sits in. "Mom" might point to *Aging-parent obligations* (hard system), *Family alignment* (soft system), or both. The concern names a feeling; the system names what to do about it.

### The hardest category: soft systems

Soft systems often *fail* criterion 3 (external gates) because emotional regulation, partner alignment, and identity work don't have government offices behind them. They still belong on the Map because they pass the other five criteria robustly, and because the Hekswerk methodology treats nervous-system capacity as load-bearing — not optional. Soft systems are part of why this practice exists at all; clients with infinite capacity wouldn't need it.

**Rule for soft systems**: if a candidate passes criteria 1, 2, 5, and 6 (coherence, domain logic, persistence, characteristic failure mode) but is weak on 3 (external gates) and ambiguous on 4 (connectedness), include it as a soft system. Flag in the Map as such, so the visual treatment can distinguish it without devaluing it.

## 5. The gut heuristic and its complement

### The gut heuristic

*"Damn, that's a lot to hold."*

This is real. It's the felt-sense version of criterion 1 (coherence) and criterion 5 (persistence) firing together — the cognitive load of a system that has many parts AND extends over time.

### Its blind spot

Light systems. Things that don't feel heavy week to week but are unmistakably systems over the arc. Language is the prototype. Long-term medical maintenance is another. Slow-burning systems that the gut underestimates.

### The complement test

*"Is there a category of professional specialist who handles this?"*

If yes, it's almost certainly a system, because the market only specializes around real clusters. Specialists exist when the underlying domain has enough internal complexity, external gates, and characteristic failure modes to support a professional category.

Quick scan:

- Immigration attorneys exist → visa is a system.
- International credential evaluation services exist → credential recognition is a system.
- Expat tax CPAs exist → cross-border tax is a system.
- Pet relocation companies exist → pet transport is a system.
- Relocation coaches exist (this is you) → the meta-relocation itself is a system, which is what Hekswerk is selling.

Run both tests in parallel. The gut catches the heavy ones; the complement catches the quiet ones.

## 6. The adjacent vocabulary

A system is bigger than:

- **A task**: discrete action. "Renew passport."
- **A decision**: choice point. "Sell or rent the house?"
- **A concern**: worry or focus area. "What about my mom?"

Tasks and decisions live *inside* systems. Concerns sometimes name systems and sometimes name fragments.

Part of the Phase 1 work is translating *concerns* (how clients talk) into *systems* (how the methodology operates). When a client says "I'm worried about my mom," the methodology asks: what system does that worry live in? Usually two: aging-parent obligations (hard system) and family alignment (soft system).

Resist treating concerns as their own category on the Map. They are the input; the systems are the output.

## 7. Granularity rules

When does a sub-cluster get promoted to its own system vs. staying nested inside a parent?

**Promotion criterion**: if the sub-cluster independently passes 3+ of the six criteria, it earns its own slot on the Map.

**Examples**:

- *Housing* splits into *origin disposition* and *destination acquisition*. Both independently pass 5+ criteria. → Two systems, not one.
- *Pet transport* could theoretically split into *vet certifications* and *transit method*, but neither independently passes 3+. → One system at the parent level.
- *Language* could split into *speaking* and *bureaucratic literacy*, but the split fragments the domain logic and the external actors are largely the same. → One system.

**When in doubt**: prefer fewer, larger systems for the first Map draft. Promoting later is cheap; demoting in front of a client is awkward.

## 8. How to use this in practice

**During Phase 1 intake**: Don't run the checklist out loud. Listen for what the client raises and notes how they cluster it. Mental note when something pings the criteria. The checklist is a tool for you, not a script for the conversation.

**During Systems Map drafting (Step 3 of D-01 production)**: Run the checklist explicitly for each candidate. For borderline cases, write the score (e.g., "passes 4/6 — included as system; weak on external gates because soft") in your working notes for future reference.

**During the walkthrough call with the client**: Don't show them the checklist. Trust the work. If they question why something is or isn't a system, explain in plain language: "this has its own logic, its own external actors, and its own ways of going wrong — that's why I treated it separately."

**Across engagements**: Track which candidates were borderline and how they resolved in practice. Update the Systems Library (D-01 Section 5) when patterns emerge — both new systems to add and granularity adjustments to existing ones.

## 9. Open questions (v0.1)

- **Criterion weighting.** Currently all six criteria count equally. Are some more important? Suspicion: failure mode is the strongest predictor of system-hood; connectedness is the weakest, because almost everything connects to something at low resolution.
- **The 3/5 threshold.** Pulled from intuition, not data. Track actuals across the first 3–5 engagements and revise if needed.
- **Soft system criteria adjustment.** The rule of "pass 1, 2, 5, 6; accept weakness on 3 and 4" is provisional. May need refinement once more soft systems have been mapped in practice.
- **Cultural/destination variation.** These criteria are written from a US→Western Europe perspective. Other destinations may have systems that don't fit cleanly — e.g., places where bureaucracy is informal rather than formal, in which case the "external gates" criterion needs to flex.
- **Personal-stage variation.** A first-time relocator's Map may legitimately have more systems than an experienced one's, even with the same household composition. Capacity for absorption is a hidden variable that may belong in the criteria somehow. Worth tracking.
- **The "concern → system" translation move.** Sometimes a single concern points to multiple systems (the "mom" example). The rule for handling those translations cleanly during intake notes may want its own short methodology note (M-02?) eventually.
